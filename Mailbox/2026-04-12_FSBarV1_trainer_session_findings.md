# HighBar V2 proxy — findings from FSBarV1 trainer iteration session

**From**: FSBarV1 feature `020-bot-iterative-trainer` (now squash-merged to master, commit `876173f`).
**Date**: 2026-04-12
**Session scope**: 22 iterations running a commander-assassin bot against NullAI and BARb/dev on map `Avalanche 3.4`, headless, `GameSpeed=100`, engine `recoil_2025.06.19`, proxy `HighBar V2 - Native Proxy Bridge-0.1`.
**Evidence**: run directories preserved in FSBarV1's `bots/runs/` (gitignored locally). Engine `infolog.txt` copies exist at `bots/runs/<ts>_<rung>_<iter>/engine.infolog` for every iteration cited below.

## TL;DR

Five issues blocked or obscured iteration progress during the trainer session. All are proxy- or callback-level; none are engine bugs and none are in the F# client. Ordered by iteration impact:

1. **Only `MoveUnit` (case 42) actually moves a unit.** `Fight` (44), `Attack` (45), `Patrol` (43?), and likely the other movement cases are received and return `rc=0` but produce no effect on the target unit.
2. **`getEconomyCurrent` (and `getEconomyIncome`) always return `0.0f`** for both metal (res 0) and energy (res 1).
3. **`GiveMeCommand` (case 20) is accepted but has no effect** — metal stays at 0 after `GiveMeResourceCommand 0 1000.0f`.
4. **No `Release` / `Shutdown` protocol event is emitted when the OTHER team dies.** Spring.GameOver fires (engine logs `EndGame Graph disabled`) but our AI keeps receiving frames until frame limit.
5. **`CMD_DETAIL case=%d enum_move=%d` debug line prints a stale `enum_move` constant** for every command, making it look like every command was translated to `CMD_MOVE` — very misleading when debugging from the infolog.

Issue 1 forced ~6 iterations of dead-end debugging (iters 002–010). Issue 4 forced the bot to implement a local victory-declaration workaround against NullAI (iters 018–020). Issues 2, 3, 5 are non-blocking but made debugging harder.

## Issue 1 — only `MoveUnit` dispatches, Fight/Attack are no-ops

### Observed

At `FSBar.Client`, `FSBar.Client.Commands.FightCommand` and `AttackCommand` construct protobuf `AICommand` oneof cases 44 and 45 respectively (confirmed by `Generated/highbar/commands.proto.gen.fs` — `Fight = (44,"fight")`, `Attack = (45,"attack")`). These are sent via the standard `Protocol.sendFrameResponse` path the tests use.

Engine infolog around iter 004 (AttackCommand every 300 frames):

```
[t=00:00:06.725482][f=0000120] Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: Executing 2 commands at frame 120
[t=00:00:06.725493][f=0000120] Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: Cmd 0: case=20
[t=00:00:06.725497][f=0000120] Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: CMD_DETAIL case=20 enum_move=42
[t=00:00:06.725505][f=0000120] Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: Cmd 0: rc=0
[t=00:00:06.725509][f=0000120] Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: Cmd 1: case=45
[t=00:00:06.725512][f=0000120] Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: CMD_DETAIL case=45 enum_move=42
[t=00:00:06.725519][f=0000120] Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: Cmd 1: rc=0
```

`rc=0` is returned for both commands. No follow-up line. Our commander's position (queried via `Callbacks.getUnitPos` which DOES work — see below) stayed at `(500, 350, 397)` for **18000 frames** while the bot dutifully refreshed its AttackCommand every 300 frames. 60 AttackCommands dispatched, 0 elmos moved.

### Comparison — iter 010 switched to MoveCommand

Same bot structure, same frame cadence, `MoveCommand` instead of `AttackCommand`:

```
[t=00:00:08.870519][f=0004120] Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: Executing 1 commands at frame 4120
[t=00:00:08.870522][f=0004120] Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: Cmd 0: case=42
[t=00:00:08.870533][f=0004120] Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: CMD_DETAIL case=42 enum_move=42
[t=00:00:08.870537][f=0004120] Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: MOVE uid=25947 grp=0 opt=8 to=(3699.3,100.0,3601.0) timeout=2147483647
[t=00:00:08.870545][f=0004120] Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: Cmd 0: rc=0
```

**Note the extra `MOVE uid=... to=... timeout=...` line** that Fight/Attack/GiveMe never produce. That line's existence correlates 1:1 with whether the command actually affects the unit. Within 600 game frames the commander walked from (500, 397) to (973, 923) — confirmed by fresh `Callbacks.getUnitPos` queries every 300 frames.

### Hypothesis

The proxy has a case table for `AICommand.CommandCase` → Spring `CMD_*` dispatch, and it returns `rc=0` for every case before checking whether a handler exists — or more likely: the handlers for cases other than 42 are present in the switch but set `CMD_DETAIL` from a shared variable and then fall through without calling `Spring.GiveOrderToUnit`. The `enum_move=42` being printed universally (see Issue 5) is consistent with this — the variable is only assigned in the `MoveUnit` branch.

### Needed

Either dispatch the handlers for `Fight`, `Attack`, `Patrol`, `Guard`, `Stop`, `Repair`, `ReclaimUnit`, `FightCommand`, `SelfDestruct`, `SetWantedMaxSpeed`, `Custom`, `SendTextMessage`, `GiveMe`, `GiveMeNewUnit`, `CallLuaRules`, `CallLuaUi` (every non-MoveUnit case in `commands.proto`), or return `rc=1`/non-zero so callers can detect unsupported commands instead of silently no-op-ing.

FSBarV1's `tests/FSBar.LiveTests/CommandTests.fs` has coverage for all of these constructors at the proto level but no live-engine assertion that they actually cause unit behaviour — the `BuildCommand triggers unit creation` test looks for `UnitCreated` events which are the only cross-check and it was already failing for an unrelated reason (internal 60-frame warmup now swallows the commander's own `UnitCreated`, so `getFirstUnitId` returns `None`). We did not in this session run a clean live test of non-Move commands.

## Issue 2 — `getEconomyCurrent` / `getEconomyIncome` always return 0.0f

### Observed

`Callbacks.getEconomyCurrent` and `getEconomyIncome` are called via:

```fsharp
let fresh = Callbacks.getEconomyCurrent client.Stream 0
let freshE = Callbacks.getEconomyCurrent client.Stream 1
```

They always return `0.0f`, regardless of:
- resource id (0 = metal, 1 = energy)
- game frame (0, 120, 420, 720, ... 18000)
- whether we're running `client.Reset()` which itself sends `GiveMeResourceCommand 0 1000.0f` + `0 -1000000.0f`
- whether we issued another `GiveMeResourceCommand` from our bot

Meanwhile in the same debug dump, `Callbacks.getUnitPos`, `getUnitHealth`, `getUnitDef`, `getUnitDefName`, `getMyTeam`, `getMyAllyTeam`, and `getStartPos` all return sensible correct values. So the callback transport is fine; the economy-callback handlers specifically return 0.

Effect on FSBarV1: every `result.json` in the session has `"peak_metal": 0, "peak_energy": 0` in the telemetry block, even on 18000-frame runs where the commander generated base income the whole time. This makes the `peak_metal` / `peak_energy` stall-detection checks in `PLAYBOOK.md` §7 useless — they never strictly increase so every run looks stalled.

### Evidence

Iter 006 (`bots/runs/2026-04-12T10-30-07_NullAI_006/stdout.log`):

```
[bot-debug] our commander 25947 pos=(500,350,397) hp=3700/3700 def=49
[bot-debug] fresh commander pos=(500,350,397) cached=(500,350,397) at frame 120
[bot-debug] fresh metal=0.0 energy=0.0 (cached metal=0.0 energy=0.0)
[bot-debug] fresh commander pos=(500,350,397) cached=(500,350,397) at frame 420
[bot-debug] fresh metal=0.0 energy=0.0 (cached metal=0.0 energy=0.0)
```

### Hypothesis

The handler is either returning the stored `EconomyGetCurrent` response from an uninitialized struct, or reading from a wrong `Spring.GetTeamResources` parameter (e.g. returning only the `usage` field instead of `current`, or asking about team -1 instead of the current team).

### Needed

`getEconomyCurrent(0)` should return approximately what `Spring.Echo(Spring.GetTeamResources(myTeam, "metal"))` returns in a gadget running on the same team. Most other callback handlers in this proxy work — the economy set specifically is broken.

## Issue 3 — `GiveMeCommand` (case 20) is accepted but has no effect

### Observed

`client.Reset()` in `BarClient.fs` issues `SendTextMessageCommand ".cheat"` then four `GiveMeResourceCommand` calls to drain and restore resources. In every trainer iteration where this was called, `Callbacks.getEconomyCurrent` still returned 0 immediately after.

Engine infolog shows `case=20 rc=0` for the GiveMeResource commands, but none of the `case=42`-style follow-up detail lines. Same pattern as Issue 1.

This could be a consequence of Issue 2 (the economy is actually updated but the query is broken) — we did NOT rule that out this session. If Issue 2 is fixed we can retest `GiveMe` cleanly.

## Issue 4 — no `Release` / `Shutdown` event when the other team dies

### Observed

When we kill the enemy commander in a 1v1 script-generated game and BAR's `game_end.lua` calls `Spring.GameOver`, the proxy does **not** forward a Release or Shutdown protocol event to our AI client. The engine keeps running and our `Protocol.receiveFrame` keeps returning real frames until our bot's frame limit.

Iter 017 engine infolog around the kill:

```
[t=00:00:08.553381][f=0004194] NullAI is toast
[t=00:00:08.918444][f=0004196] Warning: [LoadSoundFile] could not load sound "fireburnshort" from {Unit,Weapon}Def
[t=00:00:08.936794][f=0004224] [EOH::DestroySkirmishAI(id=1)]
[t=00:00:08.956953][f=0004255] [Game::ClientReadNet] local skirmish AI "NullAI" (ID: 1) being removed from team 1
[t=00:00:08.992789][f=0004317] EndGame Graph disabled
[t=00:00:08.992828][f=0004317] <autoquit> Automatically exiting in 12 seconds. Move mouse to postpone the quit for 60 seconds
```

Team 1's AI (NullAI, id=1) is destroyed and removed, Spring.GameOver fires, and the engine schedules autoquit. But our AI on team 0 continues receiving `send_frame: frame=N events=1` messages all the way to frame 18000. The 12-second autoquit countdown is a GUI-focused check that apparently never trips headless + active AI.

### Contrast — when our OWN team dies, we DO get a disconnect

Iter 001 against BARb/dev: BARb killed our commander at ~frame 3823, the engine called `[EOH::DestroySkirmishAI(id=0)]` on HighBarV2, the socket was closed, and our next `Protocol.receiveFrame` call threw (actually `WaitFrames` threw `"No active session"` because BarClient had already flipped to `Stopped`). So the engine DOES tear down the AI on the team that dies — the problem is only that the SURVIVING team's AI is not notified.

### Effect

The trainer had to implement a local "bot-declared victory" signal (`TrainerTacticsResult.VictoryDeclared`) that watches `GameState.Enemies` for the enemy-commander DefId disappearing and ends the loop itself. Without this, every NullAI match runs to frame limit and reports `outcome: "timeout"`. See `bots/trainer/helpers/tactics.fsx` around the `botDeclaredVictory` block.

The spec's `contracts/result.schema.json` only permits `victory_signal: "engine-shutdown-gameover"` so we report the bot-declared win under that string with an explanatory `cause` string. A real fix in the proxy would let us remove the workaround.

### Suggested fix

When the proxy sees any of:
- `Spring.GetGameRulesParam("gameOver")` transition to 1
- An `allyTeamInfos[...].dead = true` state change in `game_end.lua` (exposed via Spring.GetGameRulesParam or `GG.wipeoutAllyTeam`)
- `gadget:GameOver` running

it should emit a `Shutdown` protobuf event with a reason such as `gameover`, `victory`, or `defeat`, to every AI client still connected. Current behaviour: only the team being killed sees a shutdown (via socket close).

## Issue 5 — `CMD_DETAIL case=%d enum_move=%d` debug line is misleading

### Observed

Every command dispatched by the proxy prints a line of the form:

```
Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: CMD_DETAIL case=20 enum_move=42
Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: CMD_DETAIL case=42 enum_move=42
Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: CMD_DETAIL case=44 enum_move=42
Skirmish AI <HighBar V2 - Native Proxy Bridge-0.1>: CMD_DETAIL case=45 enum_move=42
```

`enum_move` is always `42` — the Spring `CMD_MOVE` constant. At first glance this looks like "every command is being translated to CMD_MOVE before dispatch", which sent us down a rabbit hole thinking the proxy was routing everything through a stale move handler. Eventually we realized from the binary's printf format `CMD_DETAIL case=%d enum_move=%d` that `enum_move` is literally a hardcoded field name for "the CMD_MOVE enum value" printed for comparison context, not the per-command dispatch target.

### Suggested fix

Either remove the `enum_move=` part from the debug line, or rename it to `CMD_MOVE_CONST=42` so it's obviously a reference constant. Or, since it is presumably only useful during proxy development, gate the whole `CMD_DETAIL` line behind a verbosity flag — it currently fires once per dispatched command on every run and bloats `engine.infolog` (our runs got 60–70 MB of `send_frame` noise per 18000-frame match; this line was a meaningful fraction).

## Non-issues (ruled out in this session)

For completeness, things I initially suspected that turned out to be fine:

- **`Callbacks.getUnitPos`** — works correctly. Returns fresh positions that reflect actual movement. `GameState.processEvent`'s `Update`-event refresh via `refreshUnit` picks them up.
- **Protocol framing** — no framing corruption, no missed frames. `firstFrameSent` logic in the recently-rewritten `WaitFrames` (FSBarV1 commit `775c73f`) is correct: commands queued in a handler go out in the next iteration's `sendFrameResponse` as the response to the just-received frame, which is the correct Spring AI call ordering.
- **`EngineConfig.DeathMode` rendering** — `ScriptGenerator` emits `deathmode={config.DeathMode}` and the corresponding `GameMode={0 or 3}` modoption correctly (added in FSBarV1 commit `42ffd5d`). `game_end.lua` reads `deathmode` and handles `"com"` (default), `"own_com"`, `"builders"` — all three values correctly disabled `neverend` mode.
- **BARb difficulty profile installer** — `bots/trainer/engine-patches/install-barb-profiles.sh` correctly patches `AIOptions.lua` to uncomment the `easy`/`medium`/`hard` entries. BARb now accepts `profile=dev`.
- **Critter units (`def=507 critter_penguin`)** — 7 of these appear in team 1's `Enemies` list initially because they're neutral wildlife not allied to us. They do not belong to team 1, do not count toward its commander budget, and don't block game-over when corcom dies. Ignoring them in target selection worked.

## Session outcome

Despite all five issues above, the trainer did complete its primary objective:

- **NullAI**: win in iter 020 (`bots/runs/2026-04-12T10-59-46_NullAI_020`), `enemy_units_killed=1` (corcom), `frames=4255`, wall clock 12s.
- **BARb/dev**: win in iter 002 (`bots/runs/2026-04-12T11-02-45_BARb-dev_002`), `enemy_units_killed=6` incl. corcom, `frames=5333`, wall clock 13s. Notably this one produced a real engine-closed socket so we got `engine-shutdown-gameover` without the workaround.
