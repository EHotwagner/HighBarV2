# 035 research + FR-014 cleanup: INTERNAL_ORDER removed from HighBar.Client

**Date**: 2026-04-19
**From**: HighBarV2 (Recoil proxy + F# client)
**To**: FSBarV1

## TL;DR

- **`HighBar.Client 0.1.10`** removes the `INTERNAL_ORDER` option bit that every unit-command builder was setting. If you were copying that pattern, stop — it was wrong. New builders default `Options = 0u`.
- Research branch `035-research-dispatch-paths` landed a fully-cited root-cause trace for the 034 silent-drop (`rc=0` with no nanoframe). Outcome is **(b) docs-only** — the silent reject is sim-side, past `CAICallback::GiveOrder`'s `clientNet->Send()`.
- Follow-up investigation `036` will pick up from `CCommandAI::GiveCommand` / `CBuilderCAI` to explain why `CMD_BUILD` for a live own-team commander is accepted at the network layer but never materialises in-world.

## API surface change

`clients/fsharp/src/Commands.fs` no longer sets any `Options` bit on unit commands. Every builder (`MoveCommand`, `BuildCommand`, `PatrolCommand`, `AttackCommand`, `GuardCommand`, `StopCommand`, `RepairCommand`, `ReclaimUnitCommand`, `FightCommand`, `SetWantedMaxSpeedCommand`, `SelfDestructCommand`, `CustomCommand`, `SetOnOffCommand`) now sends `options=0` on the wire.

No public signature changes. The private `INTERNAL_ORDER = 8u` literal is deleted.

## Why the cleanup

At Recoil `1770ea0` (`rts/Sim/Units/CommandAI/Command.h:81–94`), bit-3 is `INTERNAL_ORDER`, a real engine flag that distinguishes orders generated by the engine's own auto-behaviour (auto-repair, auto-reclaim, BuilderCAI fight/patrol continuations) from orders submitted by a player or AI client. Setting it from a client AI meaningfully alters sim behaviour:

- `FactoryCAI.cpp:344` — commands flagged internal skip the repeat-order queue push, so a build in repeat-mode silently won't repeat.
- `BuilderCAI.cpp:759, 772, 1026, 1136, 381` — several reclaim / resurrect / target-intercept branches take different paths for internal orders.

None of these caused the 034 silent-drop (wrong layer), but they were behavioural drift we didn't want. See `specs/035-research-dispatch-paths/findings.md §9` for the full quote-trail.

## 035 outcome in one paragraph

Both `Engine_handleCommand` and `Engine_executeCommand` funnel into the same default-case dispatch at `SSkirmishAICallbackImpl.cpp:1143`, which branches on the `S*Command.unitId` field: `≥ 0` → `CAICallback::GiveOrder` (real network send + return 0), `< 0` → `CAICallback::GiveGroupOrder` (a `return 0;` stub — silent drop). Our failing test (T3.3) passes a `unit_id ≥ 1` obtained from `UnitCreated` callback, so it hits the happy path to `clientNet->Send()` and the rc=0-no-effect happens past that point, in sim-side `CCommandAI::GiveCommand` / `CBuilderCAI`. Scope of 036.
