# Implementation Plan: Comprehensive Gamestate Snapshot

**Branch**: `032-batch-callback-rpcs` | **Date**: 2026-04-19 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/032-batch-callback-rpcs/spec.md`

## Summary

Add a single `getGameState()` callback that returns, in one protobuf round-trip, all friendly units, all visible enemies (LOS + radar-only), and the current economy record. The proxy enumerates the unit sets itself (friendlies via `getTeamUnits`, enemies via `getEnemyUnitsInRadarAndLos`) and populates per-unit fields (position, health for LOS, unitDefId, team) using the existing `SSkirmishAICallback` function table — no engine modifications. Per the clarified user choice, the mechanism is a **batched query** (one request, one response) rather than async pipelining.

## Technical Context

**Language/Version**: C11 (proxy), F# / .NET 10.0 (client)
**Primary Dependencies**: protobuf-c (C), Google.Protobuf 3.28 (F#), Recoil `SSkirmishAICallback` (read-only)
**Storage**: N/A — in-memory snapshot only, no persistence
**Testing**: xUnit (F# unit + integration + persistent live-engine), CTest/CMake (C proxy)
**Target Platform**: Linux `.so` proxy + cross-platform F# client; validated against spring-headless 2025.06.19
**Project Type**: Library (proxy shared-lib + F# client lib) — matches existing repo layout
**Performance Goals**: Full snapshot < 10 ms wall-clock at 200 friendlies + 50 enemies; IPC round-trips per tick = 1
**Constraints**: Constitution latency budget (<1 ms proxy overhead, <500 us socket round-trip). Bounded message size (default 4 MB). Arena allocation on hot path. No heap allocation per event.
**Scale/Scope**: Reference load 200 friendlies + 50 enemies; must degrade gracefully (still one call) up to engine `MAX_UNITS`.
